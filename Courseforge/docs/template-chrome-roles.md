# `data-cf-role="template-chrome"` — page-chrome marking convention

**What it does.** Courseforge marks repeated page chrome (header, footer, skip-link) with the attribute `data-cf-role="template-chrome"`. Downstream consumers — Trainforge's HTML text extractor in particular — check for the attribute and skip the subtree when building chunk text. Result: per-chunk boilerplate like `© 2026 SAMPLE_101. All rights reserved.` never lands in `chunk.text`, never lands in training-pair prompts, never lands in retrieval indexes.

**Why it matters.** Before this convention, the n-gram boilerplate detector in `Trainforge/rag/boilerplate_detector.py` was the only line of defense. It works on *corpus-wide repetition above a threshold* (see `DEFAULT_MIN_DOC_FRAC = 0.30`). Footers that appeared on every page of a small corpus missed the threshold; novel template chrome (new course styles, different footer copy) never triggered. The `data-cf-role` convention is a structural signal that doesn't depend on corpus size, threshold tuning, or n-gram content. The n-gram detector stays as belt-and-suspenders for IMSCC packages not generated by Courseforge.

## Where Courseforge emits it

In `Courseforge/scripts/generate_course.py::_render_page_shell` (the function that wraps every generated HTML page), three elements carry the attribute:

```html
<a href="#main-content" class="skip-link" data-cf-role="template-chrome">Skip to main content</a>
<header role="banner" data-cf-role="template-chrome">
  <p>COURSE_CODE — Week N</p>
</header>
<main id="main-content" role="main">
  <h1>Page title</h1>
  <!-- actual content — NOT chrome-marked -->
</main>
<footer role="contentinfo" data-cf-role="template-chrome">
  <p>© 2026 COURSE_CODE. All rights reserved.</p>
</footer>
```

`<main>` is deliberately NOT marked — that's the page content container, and its text is what Trainforge needs to extract.

## How Trainforge skips it

In `Trainforge/parsers/html_content_parser.py::HTMLTextExtractor`, a depth counter tracks `data-cf-role="template-chrome"` ancestors. When the counter is nonzero, `handle_data` returns early:

```python
def handle_data(self, data):
    if self.in_script or self.in_style:
        return
    if self._template_chrome_depth > 0:
        return
    text = data.strip()
    if text:
        self.text_parts.append(text)
```

The counter decrements on close tags for a known small tag set (`header`, `footer`, `a`, `div`, `nav`, `aside`) — the tags Courseforge uses for chrome. Keeping this set narrow avoids spuriously decrementing on unrelated close tags when chrome is nested.

## What values are reserved

| `data-cf-role` value | Meaning | Trainforge behaviour |
|---|---|---|
| `template-chrome` | Repeated page chrome (header, footer, skip-link) | Skip subtree in text extraction |
| *(anything else)* | Reserved. Not used today; extensions require a coordinated PR touching both Courseforge emit and Trainforge consume. | Not defined — fall back to content extraction |

Downstream consumers should **not** overload this attribute with values other than `template-chrome` without a coordinated Ed4All-wide PR. The namespace is under `https://ed4all.dev/ns/courseforge/v1` (same as other `data-cf-*` attributes).

## What this doesn't do

- **Doesn't remove chrome from the rendered HTML.** Browsers and LMSes still see it, screen readers still read it — the attribute is purely a signal to text-extraction pipelines.
- **Doesn't replace the n-gram boilerplate detector.** That stays in place as belt-and-suspenders for non-Courseforge IMSCC packages. Its `footer_contamination_rate` metric stays meaningful on corpora without `data-cf-role` markers.
- **Doesn't mark data-cf-component elements as chrome.** Interactive components (flip-cards, self-checks, activity cards) are content, not chrome — they carry `data-cf-component="…"` and are extracted normally.
