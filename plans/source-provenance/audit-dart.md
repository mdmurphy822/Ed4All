# DART Source-Provenance Audit

Scope: audit the DART subsystem for source-provenance emission quality ahead of provenance-v0.3.0 work. Read-only.

## Current state — what DART emits today

### Synthesized JSON sidecar (`*_synthesized.json`)

Produced by `multi_source_interpreter.batch_synthesize_all()` at
`DART/multi_source_interpreter.py:1246` and on the single-file path at
`convert_single_pdf()` (`DART/multi_source_interpreter.py:1278`). The top-level
shape is:

```json
{
  "campus_code": "ADI",
  "campus_name": "Adirondack Community College",
  "sections": [ { ... per-section record ... } ]
}
```

Per-section record shape (emitted by `auto_synthesize_section()` at
`DART/multi_source_interpreter.py:638`):

```json
{
  "section_type": "contacts",
  "section_title": "Campus Contacts",
  "data": { ... type-dependent payload ... },
  "sources_used": { "structure": "pdfplumber table headers",
                    "content": "pdftotext entity matching" }
}
```

`sources_used` is the only provenance hint today. It is a **static, hand-coded
string per section type** (see the branches at lines 647–699). It describes
which pipeline stage was consulted for the section **as a whole**, not which
source produced any given field.

Type-specific `data` payloads:

- `campus-info` / `credentials` / `roster` → `{"pairs": [[key, value], ...]}`
- `contacts` → `{"contacts": [{name, phone, email, title, notes}, ...]}`
  (no `source` field on each contact — see `synthesize_contacts`,
  `DART/multi_source_interpreter.py:481`)
- `systems` → `{"headers": [...], "rows": [{label, student, faculty}, ...]}`
- prose (`no-account`, `guest`, `overview`) →
  `{"paragraphs": [str, ...]}`

### Quality report sidecar (`*.quality.json`)

Only written on the single-file path (`convert_single_pdf`, line 1306). Built by
`build_quality_report()` at `DART/multi_source_interpreter.py:997`. Shape:

```json
{
  "extraction_sources": ["ocr", "pdftotext", "pdfplumber"],
  "table_quality": { "contacts_p3": "complete", "systems_p5": "headers_only" },
  "entity_counts": { "phones": 4, "emails": 4, "urls": 6, "names": 12 },
  "wcag_validation": { "compliant": true, "critical_count": 0, "issues": [...] },
  "content_profile": { "pedagogical_pattern": "...", "concept_count": ... },
  "confidence_score": 0.873,
  "section_count": 7
}
```

This is the closest thing DART has to structured provenance. But it is
**document-level aggregate**: `extraction_sources` is the set of stages used
anywhere in the document; `table_quality` is keyed by `{section_type}_p{page}`
so it crosses section + page but is not reachable from a rendered block.

### Combined input JSON (`*_combined.json`)

Read by `export_section_contexts()` (line 339). Each source already carries
page numbers upstream:

- `combined["sources"]["pdftotext"]`: flat string, **no page markers** (form
  feeds are stripped by `clean_text()` at line 113).
- `combined["sources"]["tables"]`: list of dicts each with `{page, index,
  headers, rows, num_cols, num_rows}` (`page` is preserved, see line 287).
- `combined["sources"]["ocr_pages"]`: list of `{page, ocr_text}` (page
  preserved).

`build_section_context()` (line 275) attaches a coarse `page_range: (start,
end)` tuple per section via `estimate_page_range()` (line 259). The mapping is
**hard-coded per `section_type`** (a 7-campus fixture table assuming ~6-page
PDFs). This range is used only at context-build time and is **dropped from
the synthesized output** — `auto_synthesize_section` does not copy it into the
per-section record.

### HTML output (`*_synthesized.html`)

Generated by `generate_html_from_synthesized()` at
`DART/multi_source_interpreter.py:816`. Per-section structure:

```html
<section id="s3" class="dart-section dart-section--contacts" aria-labelledby="s3-h">
  <h2 id="s3-h">Campus Contacts</h2>
  <div class="contact-card dart-contact-card"> ... </div>
</section>
```

Emits only:

- `id` / `aria-labelledby` (accessibility wayfinding, not provenance)
- `class="dart-section dart-section--{type}"` — a *section-type* marker, not a
  source marker
- `class="dart-contact-card"` / `.dart-table--systems` / `.dart-table--info` —
  layout hints

The legacy `claude_processor` path in `pdf_converter/converter.py` emits
`<section id="..." aria-labelledby="...">` with no `dart-*` classes at all
(confirmed against `DART/output/foundations-of-lidt_accessible.html`:29+).

## Current state — what DART does NOT emit

1. **No per-block source attribution.** A contact card does not record whether
   its `name` came from a pdfplumber table header, its `phone` from pdftotext
   regex, or its `email` from proximity matching. The matching functions
   (`match_email_by_name`, `match_phone_by_proximity`, `extract_title_near_name`,
   lines 386–478) return raw strings — no `{value, source, method}` envelope.

2. **No PDF page refs on emitted blocks.** `page_range` is computed in
   `build_section_context()` (line 281) but not propagated through
   `auto_synthesize_section()`. The contacts/roster/systems data payloads have
   no `page` field. The prose `paragraphs` list is a flat `List[str]` with
   no page anchors. Tables carry `page` in the combined input but that field is
   **not copied into the per-row output** of `synthesize_systems_table` or
   `synthesize_roster`.

3. **No confidence or rationale per block.** `confidence_score` exists only at
   document-level (line 1062). The matcher heuristics silently pick the first
   hit; when `match_email_by_name` falls back to proximity (line 425) vs. name
   pattern (line 407) vs. the alternate local-part synthesis path (line 534),
   the choice is untracked.

4. **No `data-dart-*` attributes on HTML.** Grep confirms zero `data-` attrs
   in either the multi-source or legacy renderer. Downstream Courseforge has
   no HTML-layer hook to read DART provenance from.

5. **No block identifiers.** Sections get `id="s0"`..`id="sN"` (line 853),
   but individual `<p>`, `<div class="contact-card">`, `<tr>`, etc. have no
   stable IDs. Trainforge cannot point to a specific claim-level node.

6. **Source resolution lost before render.** `render_from_synthesized()`
   (line 788) consumes only `section["data"]` — it never sees `sources_used`
   or `page_range` even if those were promoted — so any per-block data added to
   the synthesized JSON would still need a plumbing change in the renderer.

7. **Nothing at all on the legacy claude_processor path.** `ClaudeProcessor`
   returns `DocumentStructure.blocks` as `StructuredBlock`s
   (`DART/pdf_converter/claude_processor.py:40`). The dataclass has no source,
   page, or confidence field. The prompt (line 139) instructs Claude to
   preserve content but never to emit provenance. `_generate_html_from_structure`
   (`converter.py:1057`) renders pure structural HTML.

## Staging handoff

`MCP/tools/pipeline_tools.py::stage_dart_outputs` (line 213) copies into
`{COURSEFORGE_INPUTS}/{run_id}/`:

- The `*.html` file (line 249).
- A same-stem `.json` sibling if present (line 267–272). For the preferred
  multi-source path that sibling is the `*_synthesized.json` (which carries
  the coarse `sources_used` strings).
- Additionally looks for a `*_synthesized.json` twin if names don't match
  (line 275–281).
- Writes `staging_manifest.json` with run + file list (line 291).

What is **NOT copied**: the `*.quality.json` sidecar. It is produced by
`convert_single_pdf` at line 1313 but only picked up later by
`archive_to_libv2` (`pipeline_tools.py:482`). Courseforge therefore never sees
`extraction_sources`, `table_quality`, or `confidence_score`. This is a gap
independent of emission improvements.

The Courseforge inputs staging is HTML-first. Unless Courseforge explicitly
reads the `_synthesized.json` (it currently does not — see the companion
Courseforge audit), the only provenance signal that survives is whatever is
embedded in the HTML — which today is nothing.

- `DART/multi_source_interpreter.py:638` `auto_synthesize_section` — emits

- `DART/multi_source_interpreter.py:788` `render_from_synthesized` — HTML

## Proposed minimal extensions

### 1. Per-section record in synthesized JSON

Promote already-known data and standardize shape:

```json
{
  "section_id": "s3",
  "section_type": "contacts",
  "section_title": "Campus Contacts",
  "page_range": [3, 4],
  "provenance": {
    "sources": ["pdftotext", "pdfplumber"],
    "strategy": "pdfplumber_headers+pdftotext_entities"
  },
  "data": { "contacts": [ ... ] }
}
```

Rename `sources_used` → `provenance.strategy`, add `provenance.sources` as a
typed enum (`pdftotext | pdfplumber | ocr | synthesized`), propagate
`page_range` from `build_section_context`.

### 2. Per-block provenance envelope

Wrap every leaf value that came from multi-source matching. E.g. contacts:

```json
{
  "block_id": "s3_c0",
  "name":  { "value": "Jane Doe",          "source": "pdfplumber", "pages": [3], "confidence": 1.0 },
  "email": { "value": "jdoe@campus.edu",   "source": "pdftotext",  "pages": [3], "confidence": 0.8, "method": "name_pattern" },
  "phone": { "value": "518-555-0100",       "source": "pdftotext",  "pages": [3], "confidence": 0.6, "method": "proximity" }
}
```

For tables emit `row.provenance = {source: "pdfplumber", page, table_index}`;
for text fallbacks `{source: "pdftotext", method: "label_search"}`. The
matchers already know which branch succeeded — they just discard it.

### 3. HTML-layer attribution (`data-dart-*`)

Emit on every renderable block:

- `data-dart-block-id` — stable per-block id matching synthesized JSON
- `data-dart-source` — primary source (`pdftotext|pdfplumber|ocr|synthesized`)
- `data-dart-sources` — comma-joined when multi-source
- `data-dart-pages` — e.g. `"3"` or `"3-4"`
- `data-dart-confidence` — 0.0–1.0 (optional; only when <1.0)
- `data-dart-strategy` — free-form, matches the JSON strategy label

Placement: `<section data-dart-block-id="s3" ...>`, `<div class="contact-card"
data-dart-block-id="s3_c0" data-dart-sources="pdfplumber,pdftotext"
data-dart-pages="3">`, `<tr data-dart-block-id="s5_r2" data-dart-source=
"pdfplumber" data-dart-pages="5">`, `<p data-dart-block-id="s1_p0" data-dart-
source="pdftotext" data-dart-pages="1">`.

Mirrors the `data-cf-*` convention Courseforge already uses (see root
`CLAUDE.md` § Courseforge Metadata Output), so Trainforge's existing
"JSON-LD > data-cf-* > regex" priority extractor can be extended naturally.

### 4. Include `*.quality.json` in the staging copy

Add a `.quality.json` lookup alongside the existing `.json` / `_synthesized.json`
passes at `pipeline_tools.py:267–281`. The file already exists, Courseforge
just never sees it. Extend the manifest to list it separately so downstream
consumers can find it without name-guessing.

### 5. Extend `validate_dart_markers`

Add optional provenance markers (`data-dart-source`, `data-dart-block-id`)
as warning-level checks initially, promoting to critical in a later wave
once emission is stable.

## Risks / things to watch

1. **OCR-only documents.** If pdftotext yields nothing and pdfplumber has no
   tables, everything is OCR-sourced and confidence drops hard. The current
   `confidence_score` formula (line 1057) weights source breadth 30% — a
   pure-OCR doc will score ≤0.7 regardless of content quality. Per-block
   `source: "ocr"` should not itself be treated as low-confidence; add a
   separate `ocr_quality` signal (Tesseract already returns per-word
   confidences — not currently captured).

2. **Page-range accuracy.** `estimate_page_range()` is a 7-campus fixture
   assuming a 6-page template. For arbitrary textbook PDFs these estimates
   are meaningless. Moving to actual page tracking requires retaining page
   markers through `clean_text()` (which strips `\f` at line 116) — this is
   a non-trivial refactor and is a pre-req for honest per-block `pages`.

3. **Multi-block merges.** When `synthesize_contacts` synthesizes a contact
   from an email local-part (line 533, no table header present), the
   "source" is effectively *derived* rather than *extracted*. The provenance
   envelope needs a `source: "synthesized"` / `derivation: "email_local_part"`
   case to be honest about this.

4. **Confidence scoring semantics.** If we expose per-block confidence, we
   commit to a scale. Currently matcher branches have implicit priority but
   no numeric confidence. We'll need to define canonical values (e.g. direct
   table extraction = 1.0, name-pattern match = 0.8, proximity match = 0.6,
   local-part synthesis = 0.4) and document them — otherwise downstream will
   invent their own thresholds.

5. **Backward compatibility.** Existing `_synthesized.json` files have the
   old `sources_used` shape and no `provenance`. Courseforge/Trainforge
   consumers must tolerate both — follow the Ed4All precedent of an env
   flag (`DART_EMIT_PROVENANCE=1` default-off for one wave, default-on the
   next) to mirror the ONTOLOGY §12 rollout pattern.

6. **Legacy `claude_processor` divergence.** The two emit paths will drift
   further unless the refactor touches both. Since multi-source is preferred,
   a practical compromise is emitting minimal `data-dart-source="claude_llm"`
   on the legacy renderer and accepting lower provenance fidelity there.

7. **HTML size bloat.** Adding `data-dart-*` to every `<p>` and `<tr>`
   roughly doubles emitted bytes on text-dense pages. For typical campus
   sheets this is immaterial; for full-textbook conversions (foundations-of-
   lidt at 10k+ sections) it's meaningful. Consider scoping emission to
   section-level by default and block-level opt-in.

## Open questions for synthesis phase

These need cross-subsystem decision — flag for Courseforge and Trainforge
audits to weigh in:

- **Naming.** `data-dart-*` vs `data-provenance-*` vs folding under
  `data-cf-*`. Consistency with Courseforge's existing `data-cf-*` is
  attractive but conflates layers. Recommend keeping `data-dart-*` distinct.
- **Granularity.** Section-level only, or all the way down to `<p>` / `<tr>`?
  Courseforge will cite "module X derives from DART blocks A, B, C" — if A
  is only section-granular, citations stay coarse.
- **Confidence scale.** 0.0–1.0 float, a 3-value enum (`high|med|low`), or
  a method-name string? Trainforge's leak-check / assessment-quality gates
  need to know what they'll be ingesting.
- **Stable IDs.** What is the canonical `block_id` format? If it is
  positional (`s3_c0`) it is not re-run-stable; if content-hashed it matches
  the opt-in `TRAINFORGE_CONTENT_HASH_IDS` pattern. Recommend content-hashed
  by default, positional as fallback.
- **JSON-LD option.** Should DART emit a JSON-LD block-provenance sidecar
  mirroring Courseforge's `schemas/knowledge/courseforge_jsonld_v1.schema.json`
  pattern? A `schemas/knowledge/dart_provenance_v1.schema.json` would let the
  priority extractor (`JSON-LD > data-dart-* > regex`) be consistent across
  the pipeline.
- **Staging contract.** Should `stage_dart_outputs` become a first-class
  provenance handoff (always copies HTML + synthesized JSON + quality JSON +
  optional JSON-LD sidecar) and have its manifest enumerate each file's
  role? Courseforge's ingestor then reads from the manifest rather than
  guessing filenames.
